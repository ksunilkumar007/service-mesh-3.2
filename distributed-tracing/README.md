# distributed-tracing/

## What this block does

Deploys end-to-end distributed tracing for the bookinfo service mesh using
Tempo (backend), OpenTelemetry Collector (pipeline), and OCP Console UIPlugin
(UI). Traces are generated by the bookinfo Waypoint proxy, collected via OTLP,
stored in S3-compatible NooBaa storage, and visible in OCP Console under
Observe → Traces.

---

## Lab environment

```
Cluster:    local-cluster (SNO, platform: None)
OCP:        4.20
Istio:      v1.27.5_ossm ambient mode
Storage:    ODF + NooBaa (S3-compatible via ObjectBucketClaim)
Trace gen:  bookinfo Waypoint (L7 proxy, 100% sampling)
```

```
    istio-telemetry.yaml            Telemetry CR (mesh-wide trace export)
```

## Architecture

```
bookinfo pod
  |
  v (all bookinfo traffic passes through Waypoint)
bookinfo Waypoint (Envoy)
  | Telemetry CR: provider=otel-tracing, sampling=100%
  | OTLP gRPC -> otel-collector.tracing.svc.cluster.local:4317
  v
OpenTelemetry Collector (tracing namespace)
  | batch processor
  | OTLP gRPC TLS + SA bearer token -> tempo-tempo-gateway:8090
  v
Tempo Gateway (openshift mode, OAuth)
  | SubjectAccessReview -> OCP API (checks otel-collector SA permissions)
  | forwards to distributor -> ingester
  v
NooBaa S3 (ODF)  <-  trace blocks flushed every ~5 minutes
  |
  v
Tempo Querier <- OCP Console UIPlugin
  Observe → Traces (Jaeger-compatible UI)
```

### Step 3 — Tempo storage

```bash
# Create ObjectBucketClaim
oc apply -f distributed-tracing/tempo/tempo-bucket.yaml

# Wait for Bound
oc get obc tempo-bucket -n tracing -w

# Create Tempo-compatible secret from NooBaa credentials
# (NooBaa uses AWS_* key names, Tempo requires different names)
BUCKET_NAME=$(oc get cm tempo-bucket -n tracing -o jsonpath='{.data.BUCKET_NAME}')
BUCKET_HOST=$(oc get cm tempo-bucket -n tracing -o jsonpath='{.data.BUCKET_HOST}')
BUCKET_PORT=$(oc get cm tempo-bucket -n tracing -o jsonpath='{.data.BUCKET_PORT}')
ACCESS_KEY=$(oc get secret tempo-bucket -n tracing -o jsonpath='{.data.AWS_ACCESS_KEY_ID}' | base64 -d)
SECRET_KEY=$(oc get secret tempo-bucket -n tracing -o jsonpath='{.data.AWS_SECRET_ACCESS_KEY}' | base64 -d)

oc create secret generic tempo-s3-secret -n tracing \
  --from-literal=bucket="${BUCKET_NAME}" \
  --from-literal=endpoint="https://${BUCKET_HOST}:${BUCKET_PORT}" \
  --from-literal=access_key_id="${ACCESS_KEY}" \
  --from-literal=access_key_secret="${SECRET_KEY}"
```

## Lab findings — what we learned the hard way

### Finding 1 — tracing namespace must NOT be ambient-enrolled

Enrolling `tracing` in ambient (`istio.io/dataplane-mode=ambient`) breaks
Tempo in two ways:

```
Problem 1: Readiness probes fail
  Tempo uses HTTPS probes (port 3101)
  ztunnel intercepts and expects HBONE — probe times out
  All Tempo pods stuck at 0/1

Problem 2: Memberlist clustering fails
  Tempo components use port 7946 for gossip ring
  ztunnel HBONE interception breaks the connection
  Error: "connection reset by peer" on memberlist join
```

Fix: remove `istio.io/dataplane-mode=ambient` from tracing namespace,
restart all Tempo pods.

### Finding 2 — tracing namespace must NOT have istio-discovery label

Adding `istio-discovery=enabled` to the tracing namespace registers
`otel-collector` as a mesh service. The bookinfo Waypoint then routes
to it via HBONE (ztunnel tunneling).

Since tracing is not ambient-enrolled, ztunnel on the receiving end
cannot process HBONE — connection silently fails. No traces reach
the collector.

```
Symptom: otel-collector logs completely empty despite traffic
Fix: remove istio-discovery=enabled from tracing namespace
Result: Waypoint sends plain gRPC to otel-collector -> traces flow
```

### Finding 3 — NooBaa secret key names differ from Tempo requirements

NooBaa creates a Secret with AWS-compatible key names:
```
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

Tempo requires different key names:
```
bucket
endpoint
access_key_id
access_key_secret
```

Must create a separate `tempo-s3-secret` with the correct names.
Applying TempoStack with the NooBaa secret directly gives:
```
"storage secret must contain bucket field"
"storage secret must contain endpoint field"
```

### Finding 4 — TempoStack openshift mode requires gateway enabled

```yaml
# WRONG — will be rejected
spec:
  tenants:
    mode: openshift
  # gateway not enabled

# CORRECT
spec:
  template:
    gateway:
      enabled: true
  tenants:
    mode: openshift
```

Error without gateway: `openshift mode requires gateway enabled`

### Finding 5 — gateway and jaegerQuery ingress are mutually exclusive

```yaml
# WRONG — rejected
queryFrontend:
  jaegerQuery:
    enabled: true
    ingress:
      type: route     # conflicts with gateway

# CORRECT — gateway creates its own route
queryFrontend:
  jaegerQuery:
    enabled: true
    # no ingress section
```

Error: `cannot enable gateway and jaeger query ingress at the same time`

### Finding 6 — otel-collector needs explicit RBAC for Tempo openshift mode

The Tempo gateway uses OCP SubjectAccessReview to verify the caller's
SA has permission to write to the tenant. Without ClusterRole binding:
```
"rpc error: code = PermissionDenied desc = forbidden"
```

Required ClusterRole resources field must be the tenantName (`dev`),
resourceNames field must be `traces`.

### Finding 7 — enableTracing: true required in meshConfig

The extensionProvider alone is not enough. Without `enableTracing: true`
in meshConfig the Waypoint bootstrap has `tracing: {}` (empty) and
never connects to the OTEL collector.

```bash
oc patch istio default --type=merge -p '{
  "spec": {"values": {"meshConfig": {"enableTracing": true}}}
}'
```

After patching, restart the Waypoint to pick up new bootstrap config.

### Finding 8 — Waypoint restart required after meshConfig changes

The Waypoint reads tracing config at startup from the Envoy bootstrap.
Patching the Istio CR updates the configmap but does not automatically
restart the Waypoint. Must restart manually:

```bash
oc rollout restart deployment/waypoint -n bookinfo
```
